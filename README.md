## ❓ 왜 Meta(Instagram) API 연동을 하지 않았는가

본 프로젝트(Insajang)는 초기 기획 단계에서 **Meta(Instagram) API 연동을 통한 자동 업로드**를 고려했습니다. 그러나 아래와 같은 이유로 **MVP 단계에서는 Meta API 연동을 제외**하고, 콘텐츠 생성·관리 중심의 구조로 방향을 전환했습니다.

---

## 1️⃣ 외부 API 의존성으로 인한 불확실성

Meta API는 다음과 같은 제약을 가집니다.

* 비즈니스 계정 필수
* 앱 심사 및 승인 과정 필요
* 정책 변경 및 권한 제한 빈번
* API Rate Limit 및 사용 제한

이는 개인 프로젝트 및 포트폴리오 관점에서 **개발 일정과 안정성을 크게 저해**할 수 있다고 판단했습니다.

---

## 2️⃣ 핵심 문제는 "업로드"가 아니라 "관리"라고 판단

Insajang이 해결하고자 한 본질적인 문제는 단순 업로드 자동화가 아니라:

* 콘텐츠 생성 프로세스 정형화
* 게시 일정 관리
* 상태 기반 콘텐츠 흐름 관리 (DRAFT → SCHEDULED → POSTED)
* 반복 업무의 자동화 구조 설계

즉, 외부 플랫폼 연동은 **핵심 도메인이 아닌 확장 기능**으로 분리하는 것이 더 합리적이라고 판단했습니다.

---

## 3️⃣ MVP 단계에서의 기술 선택 기준

본 프로젝트는 MVP(Minimum Viable Product) 단계에서 아래 기준을 우선했습니다.

* 외부 환경에 영향받지 않는 구조
* 서버 단독으로 검증 가능한 기능
* 보안, 인증, 트랜잭션 등 백엔드 핵심 역량 집중

이에 따라 Meta API 연동은 **후속 확장 포인트**로 남기고, 내부 도메인 완성도를 높이는 방향을 선택했습니다.

---

## 4️⃣ 확장 가능한 구조로 설계됨

Meta API 연동을 제외했지만, 시스템은 다음과 같이 **확장을 고려한 구조**로 설계되었습니다.

* 콘텐츠 상태 기반 도메인 모델
* 스케줄 실행 시점 분리
* 외부 API 연동을 위한 인터페이스 분리 가능

향후 Meta API 또는 다른 SNS 플랫폼 연동 시에도 기존 도메인을 변경하지 않고 기능 추가가 가능합니다.

---

## 5️⃣ 정리

> 외부 API 연동을 하지 않은 것은 기술적 한계가 아니라,
> **MVP 단계에서의 의도적인 설계 선택**입니다.

본 프로젝트는 단기 구현이 아닌, **실제 운영 가능한 서비스 구조**를 목표로 설계되었습니다.

---

### 📌 한 줄 요약

Meta API 연동은 기술적으로 가능하나, MVP 단계에서는 핵심 도메인 완성도를 우선하여 의도적으로 제외했습니다.
